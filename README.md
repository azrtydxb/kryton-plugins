## Installing Plugins

Plugins are installed from within Kryton:

1. Open **Settings → Plugins**
2. Either paste a plugin install URL and click **Install**, or call the
   registry API (`GET /api/plugins/registry`) and install by ID with
   `POST /api/plugins/registry/:id/install` (admin scope).
3. Toggle the installed plugin on. Manage per-plugin permissions from the
   same panel.

The Kryton server fetches this repo's `registry.json` via the GitHub Contents
API on a 5-minute cache; new releases on `main` propagate automatically.

## How the Registry Works

- `registry.json` is **auto-generated by CI** from each plugin's
  `manifest.json`. Do not edit it by hand — any manual change is overwritten
  on the next merge to `main`.
- Plugin source is written in TypeScript (`.ts`). CI compiles every plugin
  with `npm run build` and **commits the built `.js` files back to `main`**
  alongside the regenerated `registry.json`. The compiled `.js` files are
  what the Kryton server downloads and runs.
- On every `push` to `main`, CI runs `validate` and, if green, the
  `update-registry` job runs `build` + `generate-registry`, then commits
  `plugins/*/server/*.js`, `plugins/*/client/*.js`, and `registry.json` via
  the `github-actions[bot]` user.

## Plugin Structure

```
plugins/your-plugin/
  manifest.json        # Plugin metadata — source of truth for registry
  server/
    index.ts           # Backend entry point (TypeScript source, optional)
    index.js           # Built artifact — committed by CI, do not edit
  client/
    index.ts           # Frontend entry point (TypeScript source, optional)
    index.js           # Built artifact — committed by CI, do not edit
  __tests__/           # Vitest tests (optional but encouraged)
```

`manifest.json` example (matches `plugins/slash-commands/manifest.json`):

```json
{
  "id": "slash-commands",
  "name": "Slash Commands",
  "version": "1.0.0",
  "description": "Type / at the start of a line to open a quick-insert menu...",
  "author": "Kryton",
  "minKrytonVersion": "2.0.0",
  "tags": ["editor", "productivity", "markdown"],
  "icon": "zap",
  "client": "client/index.js"
}
```

The `client` / `server` fields point at the **built** `.js` artifacts —
not the `.ts` sources. CI guarantees the referenced files exist.

## Adding a Plugin

1. Fork this repo and create a branch.
2. Copy `templates/plugin-template/` to `plugins/your-plugin-name/`.
3. Edit `manifest.json` with your plugin's metadata.
4. Implement your plugin in `server/index.ts` and/or `client/index.ts`.
5. Run the full local check — it mirrors CI exactly:
   ```bash
   npm run build       # compile TS → JS
   npm run typecheck   # tsc --noEmit
   npm run lint        # eslint plugins/
   npm run validate    # validate all manifests + registry shape
   npm run test        # plugin smoke tests + vitest unit tests
   ```
6. Open a PR. CI runs the same five steps; merges to `main` are gated on
   green.
7. After merge, the `update-registry` job rebuilds, regenerates
   `registry.json`, and commits both. Your plugin appears in the registry
   on the next 5-minute cache refresh of any running Kryton instance.

See [docs/CONTRIBUTING.md](docs/CONTRIBUTING.md) for detailed guidelines and
[docs/PLUGIN_API.md](docs/PLUGIN_API.md) for the full server + client API.

## Development

```bash
npm install            # install dev dependencies

npm run build          # build all plugins (TS → JS)
npm run typecheck      # type-check without emitting
npm run lint           # eslint plugins/
npm run lint:fix       # eslint with --fix
npm run validate       # validate every manifest + registry consistency
npm run test           # full test suite (plugin smoke + vitest)
npm run test:unit      # vitest only
npm run test:watch     # vitest in watch mode
npm run generate       # regenerate registry.json locally (for inspection)
```

## Branch Protection

- `main` is protected — no direct pushes; PRs only.
- CI must pass (`build → typecheck → lint → validate → test`) before merge.
- After merge, the `update-registry` job auto-commits built artifacts and
  the regenerated `registry.json` as `github-actions[bot]`.
